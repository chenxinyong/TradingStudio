## 4. 文档冲突分析

### 4.1 🔴 冲突 1：CTP 封装方案（3 份文档描述 3 种方案）

| 文档 | 方案 | 实际代码 |
|------|------|----------|
| doc 03 | 纯 P/Invoke 直接调用 CTP DLL | — |
| doc 07 | C++/CLI 混合程序集 | 存在但不完整（`clr_test.cpp`） |
| CTP接口封装方案 | C++/CLI（ChatGPT 建议） | — |
| **实际代码** | **C++ 原生桥接 → C 导出 → P/Invoke** | ✅ 当前路线 |

**结论：** 实际代码选了一条中间路线——比纯 P/Invoke 稳健（C++ 端处理虚函数回调），比 C++/CLI 简单（不依赖托管 C++）。文档没有准确描述当前实现。

**建议：** 写一份简短的 `12-ctp-bridge-actual.md`，记录实际架构决策和原因，作为后续开发的唯一参考。废弃 doc 03 和 doc 07。

### 4.2 🟡 冲突 2：存储路径约定

| 来源 | 路径格式 |
|------|----------|
| 精简架构 | `Data/{exchange}/{product}/{date}.tick` |
| doc 10 DDL | `{tradingDay}_{instrumentId}.tick` |
| doc 02 | `{DataPath}/ticks/{symbol}/{tradingDay:yyyyMMdd}_{symbol}.tick` |

**结论：** doc 02 是最早的权威定义，doc 10 延续了它。精简架构简化表述时丢失了目录层级和 TradingDay 概念。

**建议：** 以 doc 02 为准：`{dataRoot}/ticks/{symbol}/{tradingDay:yyyyMMdd}_{symbol}.tick`。精简架构文档中的路径描述更新为一致。

### 4.3 🟢 冲突 3：价格精度（实际一致，表述不同）

| 来源 | 价格类型 | 用途 |
|------|----------|------|
| TickRecord 代码 | `long * 10^7` | 二进制存储，性能优先 |
| doc 10 DDL | `BIGINT`（= `long * 10^7`） | 数据库存储 |
| 精简架构 ContractSpec | `decimal` | 合约规格表示，自然单位 |

**结论：** 不冲突。TickRecord 用 `long` 是存储优化，ContractSpec 用 `decimal` 是人类可读。只要 PriceScale 常量（10,000,000）全局一致，两个格式的转换是确定性的。

### 4.4 🟡 冲突 4：项目结构（3 种定义不一致）

| 来源 | 项目列表 |
|------|----------|
| doc 06 | Core, Data, Ctp（3 个） |
| doc 11 | Core, Ctp, CTPWrapper（隐含） |
| 精简架构 | Core, Data, Execution, Strategy, UI（5 个） |
| 实际代码 | Core, Bridge, Ctp, CtpInterop, Ct（5 个，职能重叠） |

**结论：** 精简架构是最终目标状态，但当前代码尚未按它重构。
