## Layer 0：交易的本质

### 0.1 根本定义

> 交易 = 用资金承担风险，以换取期望正收益的概率分布。

```
单笔交易收益 = 方向判断 × 仓位大小 × 价格变动 - 交易成本
总收益       = Σ(单笔交易收益) × 交易次数

方向判断 = f(信息)                              — 信息处理能力
仓位大小 = g(置信度, 资金, 风险约束)               — 资金管理能力
交易成本 = 手续费 + 滑点 + 冲击成本 + 机会成本       — 执行效率
```

### 0.2 盈利来源的唯一性推导

```
市场有效性假说 (EMH)：价格已反映所有已知信息。

EMH 的前提条件（不总是成立）：
  1. 信息已被传播和消化              → 信息传播需要时间
  2. 套利无成本                    → 套利需要资本 + 承担风险
  3. 所有参与者有经济理性            → 对冲盘/被动调仓/爆仓强平 ≠ 经济理性

→ EMH 在长尺度上成立（价格不可预测），在短尺度上不完美（存在定价时滞）

→ 盈利来源 = 信息优势 ∨ 速度优势 ∨ 结构性优势

  速度优势：    需要 μs 级延迟（HFT 专属）      → 个人不可能
  结构性优势：  做市商/对冲基金/交易所会员       → 个人不可能
  信息优势：    更快/更好地处理公开信息           → 个人唯一可能

→ 个人交易者的唯一可行路径 = 在某个细分领域积累信息处理优势
  （这解释了为什么盈利的个人交易者通常专精于少数品种/策略）
```

### 0.3 交易系统的本质

```
交易系统 = 将人力不可复现的投资能力，转化为可复现、可验证、可迭代的规则管道。

它替换的是人的三个弱点：
  信息处理上限（人 3-5 个变量 vs 系统 70+ 合约 × 多时间框架）
  决策一致性缺陷（情绪、疲劳、锚定效应）
  执行可靠性问题（犹豫、手滑、忘记止损）
```

**第一性原理推论 #1**：

> 交易系统的价值 = 可复现的 +EV 决策管道。不是回测收益有多高，而是从 idea 到验证结果有多快、实盘与回测有多一致。

## Layer 1：个人交易者的约束方程

### 1.1 用户画像（从具体到抽象）

```
个人交易者画像 = {
    技术背景:  20 年 C# 资深工程师，技术栈单纯，
    交易标的:  期货为主 + 股票辅助，
    交易时段:  上班时间不看盘（系统必须自主运行），晚上 + 周末做研究与复盘，
    资金规模:  50 万 – 500 万，
    交易频率:  中低频（分钟级到日级信号，不做高频/抢帽子），
    风险态度:  风险控制优先于收益率（这个资金量亏不起），
    运维要求:  简单（单人维护，没有运维团队），
    成本要求:  可控（个人预算，非机构预算，优先开源方案），
    基础设施:  单台云服务器或家用服务器
}
```

### 1.2 七个硬约束推导

**约束 1：上班不看盘 → 7×24 无人值守运行**

```
→ 系统进程持续运行，永不停机。不是"开盘启动、收盘关闭"，而是"激活/待命"状态切换。
→ 交易时段（日盘 09:00-15:00 + 夜盘 21:00-02:30）→ 激活态：采集、策略、执行
→ 非交易时段（午休、结算、周末、节假日）→ 待命态：维护、备份、等待
→ 时段切换基于交易日历+交易时段配置，全自动，无需外部指令
→ 异常恢复全自动：断线重连、崩溃重启、磁盘清理、分区创建
→ 报警分级：该有行情但没有 → 🔴紧急；休息时间无行情 → 🟢正常
→ 人只在晚上/周末介入：审查报告、调整策略、分析异常
```

**约束 2：50-500 万资金 → 风控是第一优先级，不是特性**

```
→ 这不是可以随意亏的小钱
→ 中低频交易也意味着单笔持仓时间长，暴露风险高
→ 保证金杠杆（期货 5-20 倍）放大了风险
→ 风控不是"有就行"，而是系统最核心的功能：
  - 每日最大亏损限制（硬编码，不可配置关闭）
  - 单品种最大仓位限制
  - 总保证金占用上限
  - 连续亏损自动暂停
  - 极端行情（涨跌停）保护
→ 风控优先级 = P0，与数据完整性同级
```

**约束 3：策略容量无关紧要**

```
→ 500 万资金 ≪ 期货主力合约日均成交额（> 10 亿元）
→ 不需要 iceberg/TWAP/VWAP 执行算法
→ 不需要冲击成本建模
→ 执行层可以极简化（限价单直接挂，市价单直接吃）
```

**约束 4：中低频交易 → 延迟完全不重要**

```
→ 信号基于 1min/5min/日线 Bar，不是基于 tick
→ 秒级甚至分钟级的执行延迟对策略无影响
→ 不需要关注网络延迟、GC 停顿、内存分配优化
→ 可以用更简单、更可靠的代码（不用追求零分配、不用 SIMD）
→ 这不是高频交易——不需要 C++，C# 的天然性能完全过剩
```

**约束 5：最稀缺的是研究迭代速度**

```
→ 晚上 + 周末 = 研究时间有限
→ 策略质量 = f(研究效率, 验证深度)
→ 限制因子 = 从 idea 到回测结果的分钟数
→ P0 指标 = 最小化这个时间
→ 研究环境必须顺手（LINQPad / Polyglot Notebook / 熟悉的 C# 工具）
```

**约束 6：单人 + 技术栈单纯 → 依赖越少越好**

```
→ 20 年 C# 工程师，.NET 生态内生产力最高
→ 不要引入需要额外学习/维护的技术栈
→ 不要 Python 做核心路径（虽然研究生态好，但多一个语言 = 多一个维护负担）
→ Python 仅做一次性数据探索，不参与数据管道和实盘路径
→ 外部服务：PostgreSQL（1 个），不要更多
→ 不要消息队列（单进程不需要）
→ 不要 ClickHouse/TDengine（多一个服务 = 多一个故障点）
```

**约束 7：成本可控 → 开源 + 低配服务器**

```
→ 云服务器：2C4G + 100GB SSD ≈ 100 元/月（足够）
→ 家用服务器：一次性投入，零月费
→ 软件全开源：.NET、PostgreSQL、AKShare（数据）
→ 不要付费数据终端（Wind 年费数万，个人用不起）
→ AKShare 免费覆盖国内期货品种规格和行情
→ CTP 行情通过期货公司免费提供（开户即有）
```

### 1.3 系统设计优先级（修正后）

| 优先级 | 要求 | 推导来源 |
|---|---|---|
| **P0** | 风控不容妥协 | 500 万资金 + 杠杆 + 上班不看盘 → 风控是全自动的最后防线 |
| **P0** | 数据完整、准确 | 回测信任度 = f(数据质量)。数据错 = 一切推导不可信 |
| **P0** | 研究迭代速度 | 最稀缺资源。晚上 2-3 小时 = 要跑完多个 idea |
| **P0** | 自主运行 | 白天人不在。自动启停、自动恢复、异常电话通知 |
| **P1** | 回测保真度 | 手续费/滑点/保证金/涨跌停建模精度决定回测可信度 |
| **P1** | 运维简单 | 单人维护，依赖最少化，日志清晰，问题可自查 |
| **P2** | 执行延迟 | 中低频策略，秒级延迟无影响 |
| **P2** | 成本可控 | 开源方案优先，服务器 ≤ 200 元/月 |
| **P3** | 股票市场支持 | 期货优先，股票架构预留 |

**第一性原理推论 #2**：

> 个人量化系统的三个核心指标，按优先级：
> 1. **风控可靠** — 500 万资金 + 杠杆 + 人不在场 = 风控是生存问题
> 2. **自主运行** — 上班不看盘 = 系统必须在无人值守时正常工作
> 3. **研究迭代速度** — 容量和延迟不重要

## Layer 2：市场微观结构推导

### 2.1 期货市场（中国）

#### CTP 协议 — 唯一路径

```
→ 中国期货市场统一使用 CTP
→ 上海期货信息技术有限公司开发
→ 覆盖六大交易所：上期所(SHFE)、大商所(DCE)、郑商所(ZCE)、中金所(CFFEX)、能源中心(INE)、广期所(GFEX)
→ 所有期货公司底层都是 CTP
→ 行情(MdApi)和交易(TraderApi)是两个独立 TCP 连接 ← 架构必须反映这个分离
```

| CTP 特性 | 系统影响 |
|---|---|
| TCP 协议（非 UDP） | 无需处理乱序/丢包；延迟 ms 级（非 μs 级）|
| 推模式（回调） | C++ 回调 → C# 委托 |
| 登录→订阅→回调 | 必须先登录成功才能订阅行情 |
| 非交易日断连 | 需自动重连逻辑 |
| 日盘/夜盘分离 | Bar 聚合必须尊重时段边界 |

#### 交易时间

```
日盘：09:00-10:15 → 10:30-11:30 → 13:30-15:00
夜盘：21:00-02:30（不同品种夜盘收盘时间不同：23:00 / 01:00 / 02:30）
结算：15:00-16:00（不可交易）
节假日：无行情

→ 每天 3-4 个独立时段，时段间有 gap
→ Bar 聚合不能跨 gap 合并
→ 交易日切换时间 = 15:00（不是午夜）
→ 夜盘 tick 归属于次日（"交易日"概念 ≠ 自然日）
```

#### 手续费 — 国内期货独有复杂性

```
三类计费模式：
  按成交金额万分比：开仓费率 × 合约价值（如 rb 万分之 1）
  按手固定金额：    每手 N 元（如 c 1.2 元/手）
  混合：           部分品种按金额+固定

四种收取模式：
  单边：   只收开仓，平仓免费
  双边：   开仓和平仓各收
  平今免： 平今日仓位不收费（上期所/能源中心部分品种）
  平今加倍：平今日仓位加倍收费（中金所股指期货）

→ 上期所/能源中心/中金所区分"平今"和"平昨"
→ 大商所/郑商所/广期所不区分
→ 回测必须精确建模 ± 手续费差异可达策略年化收益的 20-30%
```

#### 其他关键特性

```
保证金：  合约价值 × 保证金率（5%-20%），长假/极端行情临时调整
涨跌停板：±4% 至 ±10%（不同品种），停板时流动性枯竭
主力切换：每 3-6 月主力合约切换，需复权拼接 + 主力映射表
T+0：    当日可多次开平仓（与 A 股 T+1 本质不同）
```

#### Tick 数据的本质特征

CTP 每推送一条 tick = 一次行情快照。不是每笔成交，而是按固定频率（~500ms）推送的最新快照。

```
Tick 的本质：
  - Tick 是交易所内部 ~500ms 一次的行情快照（不是逐笔成交）
  - 快照之间可能发生多笔实际成交
  - Tick.LastPrice = 快照时刻的最新成交价
  - Tick.Volume = 从开盘累计到快照时刻的总成交量
  - Tick.Turnover = 从开盘累计到快照时刻的总成交额
```

**7 个关键特征（直接影响系统设计）**：

```
1. 非均匀时间间隔
   Tick 间隔不是固定的 500ms。高活跃期 100-300ms，低活跃期可能 2-3 秒。
   → Bar 闭合必须基于 tick 时间戳，不能按 tick 数量（"60 个 tick = 1 分钟"是错的）
   → 定时器兜底：低流动性时可能 30 秒无 tick，不能无限等待

2. 开盘 burst
   开盘前 1 分钟 tick 密度可达平时的 10-20 倍。
   → Channel buffer 必须有足够余量（10000 容量在这个时候发挥作用）
   → .tick 文件写入开 500 条批量 flush，不只是 50 条

3. 价格跳动（Price Jump）
   两个连续 tick 之间价格可能跳空（没有中间价成交）。
   → Bar 的 High/Low = tick 中 LastPrice 的 MAX/MIN（不需要插值）
   → 策略不能假设价格会"经过"中间价位

4. 累计值 vs 增量值
   Tick.Volume 和 Tick.Turnover 是当日累计值，不是该 tick 的增量。
   → 要得到某段时间的成交量 = Volume[end] - Volume[begin]
   → 跨交易日时累计归零 → 归零判断是正常的，不是 bug
   → CSV 导入和实盘使用相同的累计→增量逻辑

5. 集合竞价的假 tick
   08:55-08:59 和 20:55-20:59 的 tick 是指示性价格，不是真实成交。
   → 标志位 FLAG_AUCTION 标记
   → Bar 聚合不排斥这些 tick（集合竞价价格对开盘价有参考价值）
   → 策略可自行选择是否过滤

6. 重复/修正 tick
   CTP 偶尔会重发同一时刻的 tick（相同时间戳，但 Volume 可能不同）。
   → 正常现象（交易所修正或重传）
   → Bar 聚合自然覆盖：同一个 1min Bar 内，后到的 tick 更新 High/Low/Close
   → Bar 最终闭合时取最后一次的 Volume 快照

7. 不同品种 tick 密度差异巨大
   主力 rb/cu → 每秒 3-5 tick
   远月/不活跃合约 → 几秒甚至几十秒 1 tick
   → 不能假设所有品种 tick 密度相同
   → Bar 超时闭合对低流动性品种至关重要
```

**这些特征在系统中的对应设计**：

| 特征 | 设计要点 | 文档位置 |
|---|---|---|
| 非均匀间隔 | 时间驱动 Bar 闭合 + 30s 超时兜底 | 02 §4.4 |
| 开盘 burst | Channel 容量 10000, .tick flush batch 自适应 | 02 §4.2, 01 §性能验证 |
| 累计 Volume | FirstVolume 基准重置 + Delta 计算 | 02 §4.3 |
| 集合竞价 | FLAG_AUCTION bit, 策略可感知 | 02 §1.2 |
| 重复 tick | 同 Bar 内后到覆盖、幂等 | 02 §5.2 |
| 密度差异 | 按品种独立 Bar 聚合 + 超时闭合 | 02 §4.4 |

## Layer 3：数据架构推导

### 3.1 数据分层

```
Level 3: 另类数据         ← 卫星图像、供应链、社交情绪         (Phase 5+)
Level 2: 基本面/持仓      ← 库存、基差、仓单、会员持仓          (Phase 3+)
Level 1: 订单簿           ← 买卖 5 档深度 (CTP 提供)            (Phase 1)
Level 0: Tick             ← 每笔成交 P×V×T (CTP 提供)           (Phase 1)
Level -1: Bar             ← OHLCV 聚合 (从 Tick 计算)           (Phase 1)
```

```
→ Level 0 + Level 1 是 CT 可直接获取的基础数据（不可再拆的第一性原理数据层）
→ Level -1 是 Level 0 的聚合产物（可计算，不需要额外获取）
→ 系统核心 = Tick + L1 深度 + Bar 派生
```

### 3.2 数据量计算

**已知事实**：
```
活跃合约数：~70 个（期货主力+次主力）
活跃时段：~9.5 小时/天
活跃合约 tick 密度：~2-3 tick/s
单 tick 紧凑二进制大小：~80 bytes（仅 L1）或 ~200 bytes（含 5 档深度）
```

**Tick 数据量**：

```
单合约每天：≈ 3 tick/s × 3600 × 9.5 = 102,600 ticks ≈ 8 MB (紧凑)
全市场每天：≈ 70 × 8 × 60% = ~340 MB（非主力合约密度低）
全市场每月：≈ 340 MB × 22 = ~7.5 GB

实际经验修正：
  - 前 20 活跃合约贡献 ~80% tick 量
  - 远月合约可能只有 0.1-0.5 tick/s
  - 全市场每月：5-10 GB
```

**Bar 数据量**：

```
单合约每天 1min bar：≈ 500 个 × 80 bytes ≈ 40 KB
全市场每天：≈ 70 × 40 KB ≈ 2.8 MB
全市场每年：≈ 2.8 MB × 250 ≈ 700 MB
```

**结论**：

```
→ 全市场 tick 每月 5-10 GB
→ 全市场 1min bar 每年 700 MB
→ 单机 512 GB SSD 可保存 2 年 tick + 永久 bar
→ 不需要分布式存储
→ 不需要数据库集群
```

### 3.3 数据保留策略

| 数据 | 保留期 | 理由 |
|---|---|---|
| Tick（含 5 档深度）| 30 天 | 执行质量分析够用，超期的已聚合成 Bar |
| 1min Bar | **永久** | **唯一的原子 Bar**。所有更高周期（5/15/30/60min、日线）都从 1min 合成 |
| 5min/15min/30min/60min Bar | **不存储** | 从 1min 按需合成，零冗余 |
| 日线 Bar | 永久 + 异地备份 | 从 1min 合成后显式存储（高频访问，避免每次都聚合 540 个 1min bar） |
| 交易记录（订单/成交）| 永久 | 审计 + 策略复盘 |

**关键推导**：
```
→ 1min 是最细粒度的 OHLC 聚合单位
→ 5min bar = 连续 5 个 1min bar 的 OHLC 合并（O=第1个Open, H=MAX, L=MIN, C=第5个Close, V=SUM）
→ 同理：15min/30min/60min/日线都是 1min 的合并
→ 存储 1min = 永久保留全部信息。存储冗余粒度 = 浪费磁盘 + 数据一致性问题
→ 唯一例外：日线显式存储（因为查询频率最高，每次从 540 个 1min 聚合太慢）
```

## Layer 4：存储引擎推导

### 4.1 核心原则：不同数据 = 不同访问模式 = 不同存储

不能用一个引擎满足所有场景。先分析每种数据的访问模式，再选引擎。

### 4.2 Tick 存储

**访问模式**：

```
写入：高频追加，交易时段每秒 ~200 ticks（全市场）
读取：顺序扫描（回测按时间范围读 tick）
修改：从不修改
删除：整文件删除（30 天滚动）
查询：不需要跨品种 JOIN、不需要复杂 SQL（tick 总是先聚合为 Bar 再分析）
```

**候选方案**：

| | PostgreSQL | ClickHouse | 自定义二进制 |
|---|---|---|---|
| 写吞吐 | ~50K rows/s | ~1M rows/s | 磁盘带宽 |
| 读扫描 | 行存，中 | 列存，快 | O(1) 随机访问 |
| 运维负担 | 需 PG 服务 | 额外 +1 服务 | **零** |
| 查询能力 | SQL 全 | SQL 子集 | 无 |
| 崩溃恢复 | WAL | 最终一致 | Header+Footer CRC |

**推导**：

```
→ Tick 不需要 SQL 查询（总是先聚合再分析）
→ Tick 写入路径越短越好（避免 WAL/事务/索引开销）
→ Tick 保留 30 天就删，不存在长期管理问题
→ 固定记录长度 = O(1) 定位到第 N 条 tick
→ 零外部依赖（不需要安装任何数据库服务）

→ Tick 存储 = 自定义 .tick 二进制文件
  格式：Header(128B) + N×Record(固定大小) + Footer(32B)
  命名：{tradingDay:yyyyMMdd}_{symbol}.tick
  崩溃恢复：文件大小检查 + Footer CRC 验证 + 自动重建
```

### 4.3 Bar 存储

**访问模式**：

```
写入：批量追加（每天闭市后写入一次）
读取：
  - 按 symbol + 时间范围过滤 → "rb 主力 2024 全年 1min"
  - 跨品种聚合 → "所有黑色系品种日线"
  - 列选 → "只查 Close + Volume"
数据量：~1 GB/年，永久
```

**候选方案**：

| | Parquet 文件 | PostgreSQL | ClickHouse |
|---|---|---|---|
| 列式查询 | ✅ | ❌ 行存 | ✅ |
| 跨品种 SQL | ❌ 需自建 | ✅ | ✅ |
| JOIN 元数据 | ❌ | ✅（同库） | ❌ 需跨库 |
| 运维负担 | 零 | 已有（元数据同库）| 额外 +1 |

**推导**：

```
→ Bar 需要跨品种查询 → SQL 有价值
→ Bar 写入频率极低（每天一次批量）→ PG 的写入开销无关紧要
→ Bar 需要和元数据（合约信息、手续费）JOIN → 同库最好
→ 数据量 1 GB/年 → PG 行存性能完全够（不需要列存的优势）
→ 元数据本来就在 PG → 让 Bar 也进 PG = 不增加运维负担

→ Bar 存储 = PostgreSQL（与元数据同库）
  如果未来数据量增长到 100 GB+，迁移到 ClickHouse 的成本可控（IBarStore 接口隔离）
```

### 4.4 元数据 + 交易记录存储

```
元数据（合约/品种/手续费/保证金/交易日历/时段）：关系型数据，天然适合 PostgreSQL
交易记录（订单/成交/持仓）：需要 ACID（订单状态机），必须用关系型数据库

→ 元数据 + 交易记录 = PostgreSQL
  与 Bar 同一个实例，不同的表/Schema
```

### 4.6 为什么不需要 Redis

```
问题：实盘中策略反复读数据库会慢且增加 PG 负载。需要用 Redis 做缓存吗？

分析实盘中策略需要的数据：

  类型 A：实时数据（最新 tick、进行中的 bar、最新闭合的 bar）
    → 来源：CTP → Channel → BarAggregator（全在内存中）
    → 策略直接从 BarAggregator 读，不经过任何存储
    → 数据库负载：零

  类型 B：历史数据（计算指标需要回溯 N 个 bar）
    → 策略启动时：从 PG 一次性加载历史 bar 到内存（几百 KB 到几十 MB）
    → 策略运行中：新 bar 从 BarAggregator 推送到内存缓存，追加到历史窗口
    → 数据库负载：只在启动时读一次

  类型 C：元数据（合约规格、手续费率、保证金率）
    → 数据量 < 100 KB
    → 启动时从 PG 加载到内存
    → 数据库负载：只在启动时读一次

  类型 D：交易状态（订单、成交、持仓）
    → 实时维护在 ExecutionService 内存中
    → 持久化到 PG（写），但不从 PG 读

结论：
  → 所有高频访问的数据都在内存中，不经过任何数据库
  → PG 只在两种场景被访问：
    1. 启动时：加载历史数据（一次性）
    2. 写入时：持久化 Tick/Bar/Order（后台写入，不阻塞策略）
  → 不需要 Redis 做读缓存——进程内内存就是最快、最简的缓存
  → 不需要 Redis 做 Pub/Sub——单进程内 Channel 零开销

什么情况下才需要 Redis？
  → 多个进程需要共享实时数据（如 Python 研究进程需要访问实时 bar）
  → 策略崩溃恢复需要快速恢复状态（内存丢了，从 PG 重建太慢）
  → 这些是 Phase 3+ 的场景，Phase 1-2 不需要

当前决策：不需要 Redis。内存缓存 + PG 持久化 = 最简方案。
│                                                 │
│  运维负担：1 个外部服务（PostgreSQL）              │
└─────────────────────────────────────────────────┘
```

## Layer 5：回测引擎推导

### 5.1 回测的本质

> 回测 = 用历史数据模拟决策系统，回答一个问题：「如果我用这个规则在过去交易，结果会怎样？」

**回测不是**：
- ❌ 证明策略能赚钱（只能证伪，不能证实）
- ❌ 参数优化工具（那是过拟合温床）
- ❌ 一个数字（"夏普 3.0"）— 需要分布、区间、统计显著性

**回测的价值**：
```
→ 在实盘前暴露策略缺陷
→ 回答：什么市场环境下失效？最大回撤多少？恢复多久？
→ 失效模式分析 > 收益率点估计
```

### 5.2 回测与实盘的同一性（最核心的原则）

```
→ 如果回测代码 ≠ 实盘代码 → 存在"翻译误差"
→ 翻译误差 = 回测结果不代表实盘系统的行为
→ 违背回测的根本目的

架构含义：
         ┌──────────────────────┐
         │   策略信号模块         │  ← 同一份 DLL
         │   Strategy.dll       │
         └──────┬───────────────┘
                │
     ┌──────────┴──────────┐
     │                     │
  ┌──▼────────┐    ┌───────▼────────┐
  │回测 Runner │    │  实盘 Runner    │
  │           │    │                 │
  │数据: .tick│    │数据: CTP Tick   │
  │资金: 虚拟  │    │资金: 真实        │
  │撮合: 虚拟  │    │撮合: CTP/交易所  │
  └───────────┘    └────────────────┘
```

### 5.3 防过拟合

```
→ 滑动窗口验证（Walk-Forward）：
  训练 [2年] → 选最优参数 → 测试 [后6月]
  滚动前进 → 所有测试期加权评估

→ 参数敏感度检查：
  参数 ±20% → 收益率变化 > 30%？→ 不稳定，可能过拟合

→ Deflated Sharpe Ratio：
  考虑"试了多少个策略变体"，修正夏普的统计显著性
```

### 5.4 回测必须建模的成本

| 成本项 | 建模精度 | 说明 |
|---|---|---|
| 手续费 | 精确到每手 | 根据品种、开平、今昨分别计算 |
| 滑点 | 固定 + 比例 | 默认 1 tick + 万分之 0.5 |
| 保证金 | 每日变动 | 根据历史保证金率调整 |
| 涨跌停 | 成交可行性 | 停板方向不可成交 |
| 主力换月 | 价差补偿 | 在换月日做复权处理 |

## Layer 6：执行层推导

### 6.1 核心约束：人不在场

```
上班时间 = 交易时间。人只能在晚上和周末介入。

这意味着：
  开盘前：系统自动启动行情采集 + 策略引擎
  交易中：全自动运行。信号→风控→下单→成交，无需人工确认
  异常时：手机推送报警（不是发到屏幕——人不在看）
  收盘后：自动停止策略，生成当日报告
  下班后：人审查报告、异常日志、调整参数

→ 这是系统的最高可靠性要求——不是"最好能跑"，而是"必须能跑"
→ 一个白天出 bug，可能 = 一整天无法交易（机会成本）或更糟（风控失效）
```

### 6.2 执行 = 可靠的状态转换

```
信号 → 风控检查 → 生成订单 → CTP 发送 → 成交回调 → 仓位更新
```

| 环节 | 故障后果（人不在场时） | 可靠性要求 |
|---|---|---|
| 信号输入 | 全天错过信号 | 高 |
| 风控检查 | 发出危险订单 = 真金白银亏损 | **极高** |
| CTP 发送 | 订单未发出 = 策略信号落空 | 高 |
| 成交回调 | 系统仓位 ≠ 实际仓位 → 后续决策全错 | 高 |
| 断线重连 | 长时间离线 = 数据缺失 + 错过平仓 | **高** |

### 6.3 风控：针对 50-500 万资金的具体设计

```
硬性风控（硬编码，不可通过配置文件关闭）：
  1. 日内最大亏损 = 总资金的 2%（硬限制，触及即平仓+停策略+电话报警）
     → 500 万 × 2% = 10 万/天上限。亏得起，不会死。
     → 中低频交易日均交易次数少（可能 0-3 次/天），2% 是合理的。
  2. 单笔订单保证金 ≤ 总资金的 10%
     → 500 万 × 10% = 50 万/笔。一个品种不会押太多。
  3. 总保证金占用 ≤ 总资金的 40%
     → 同时持仓最多用 200 万保证金（500 万时）
     → 留 60% 资金应对不利波动，防止被追保
  4. 无效订单硬拒绝（价格≤0、数量≤0、非交易时段下单）
  5. 连续 3 笔亏损 → 暂停当日交易 → 报警
  6. 系统仓位 vs 实际仓位不一致 → 立即同步 + 暂停策略 → 报警
  7. 开盘/收盘强制检查（开盘无连接报警，收盘持仓异常报警）

软性风控（可配置参数，晚上研究调整）：
  1. 单品种最大仓位（手数或保证金占比）
  2. 是否允许隔夜持仓
  3. 日内最大交易次数
  4. 回撤超过 X% 暂停交易
```

### 6.4 7×24 无人值守运行

```
→ 系统进程持续运行，永不停机。不分"启动"和"停止"，只有"激活"和"待命"。
→ 交易时段（日盘+夜盘）= 激活态：采集行情、执行策略
→ 非交易时段（午休、结算、周末、节假日）= 待命态：维护连接、等待下一时段
→ 人永远不在。系统必须自己处理所有异常。
```

**进程生命周期**：

```
[启动] ──▶ [初始化] ──▶ [待命] ◀──────────────────┐
                          │                        │
                    (交易时段开始)                   │
                          ▼                        │
                      [激活] ──▶ (交易时段结束) ────┘
                          │
                    (崩溃/异常)
                          │
                          ▼
                      [恢复] ──▶ [待命]
```

**时段感知（Session Awareness）**：

```
系统根据 TradingSession 配置自动切换状态，无需外部指令：

  交易日判断：    TradingCalendar（非交易日全天待命）
  时段切换：      TradingSession 配置（精确到分钟）
  午休处理：      11:30→13:30 待命（CTP 可能断连，正常现象）
  结算时段：      15:00→16:00 CTP 关闭，16:00 后夜盘品种重连
  周末/长假：     连续多日待命，无报警

  关键：系统必须知道"现在是该有行情的时段"vs"现在是休息时间"
  → 该有行情但没有 → 报警（CTP 断连）
  → 休息时间没有行情 → 正常，不报警
```

**自动恢复矩阵**：

| 故障 | 检测方式 | 恢复动作 | 报警 |
|---|---|---|---|
| CTP 断连（交易时段）| 心跳超时 | 指数退避重连 | 🟡→🔴（>5 分钟） |
| CTP 断连（非交易时段）| 心跳超时 | 等待下一时段再连 | 🟢 不报警 |
| PG 断连 | 查询超时 | 重连；.tick 不受影响 | 🟡 |
| 进程崩溃 | Windows Service | 自动重启 → 恢复状态 | 🔴 |
| 磁盘空间不足 | 定时检查 | 清理过期 .tick 文件 | 🔴 |
| 内存泄漏 | 定时监控 | 预警；> 阈值自动重启 | 🟡→🔴 |

**待命态维护任务**：

```
每日收盘后（自动化）：
  1. 关闭当日 .tick 文件，写入 Footer + CRC32
  2. 聚合当日最后一批 1min Bar → 写入 PG
  3. 合成当日日线 Bar → 写入 PG
  4. 清理超过 30 天的 .tick 文件
  5. 检查 PG 分区（提前创建下月分区）
  6. 生成当日报告（交易次数、盈亏、异常）
  7. 备份日线数据

周末（自动化）：
  1. PG 全量备份
  2. 磁盘空间检查
  3. 日志轮转
```

### 6.5 报警机制

```
三级报警：

🔴 紧急（立即处理，不能等到下班）：
  - 日内亏损触及上限
  - 持仓异常（系统仓位 ≠ 实际仓位）
  - 连续下单失败
  - 风控规则被触发
  → 推送：电话通知（Pushover / Server酱 / 短信）

🟡 警告（今天下班后需要看）：
  - CTP 断线重连（但已恢复）
  - 信号生成异常（某策略报错）
  - PG 写入延迟偏高
  → 推送：App 通知

🟢 信息（日报里看即可）：
  - 今日交易汇总
  - 策略信号统计
  - 系统资源使用
  → 推送：无。日报中查看
```

## Layer 7：反馈闭环推导

### 7.1 为什么需要

> 没有反馈的系统 = 不知道自己在犯错。

```
→ 实盘 vs 回测出现系统偏差 → 需要知道并修正
→ 市场结构改变 → 策略失效需要识别并退役
→ 反馈闭环 = 持续比较预期与实际
```

### 7.2 反馈频率

```
每日：
  - 实盘收益 vs 回测同期预期（±1σ 范围）
  - 滑点实际 vs 假设
  - 信号执行率

每周：
  - 本周收益是否在合理范围？
  - 是否有异常订单？

每月：
  - 滚动夏普趋势（下降 = 策略失效）
  - 参数是否需要重估计？
  - 市场结构变化？（波动率、相关性）

策略退役标准：
  - 连续 2 月实盘显著低于回测预期（> 2σ）
  - 或滚动 6 月夏普 < 0.5
  - 或最大回撤超过历史最大回撤的 1.5 倍
```

## Layer 8：技术选型推导

> 技术选型的完整推导已整合到本文档 Layer 8。本节保留核心结论。

### 8.1 总览

| 决策 | 选择 | 关键理由 |
|---|---|---|
| 运行时 | .NET 10 | C# 20 年经验、强类型安全、Native AOT、P/Invoke 成熟 |
| Tick 存储 | 自定义 .tick 二进制文件 | 零运维、写路径最短、O(1) 定位 |
| Bar 存储 | PostgreSQL（分区表） | 与元数据同库、数据量小行存够用、SQL 查询 |
| 元数据 + 交易记录 | PostgreSQL | 关系型天然适合、ACID |
| 内部管道 | Channels + IAsyncEnumerable | 单进程零开销 |
| 消息队列 | 暂不需要 | Phase 1-2 单进程够用；Phase 3+ 需要进程隔离时加 Redis Streams |
| CTP 接入 | P/Invoke 自封装 | 需要完全控制，无黑盒 |
| 研究环境 | C# Polyglot Notebook + Python 辅助 | 策略代码与实盘同语言，Python 做一次性探索 |
| 桌面监控 | **WPF** | C# 原生，本机使用，直接连后端 |
| Web 远程监控 | **Vue 3 + SignalR** | 上班时手机/浏览器远程查看，实时推送 |
| Web API | **ASP.NET Core Minimal API + SignalR** | 为 Vue 前端提供 REST + 实时推送 |

### 8.2 为什么 WPF + Vue 两种前端

```
→ 本机场景：在交易服务器前坐着看 → WPF 桌面程序，最顺手的 C# 原生体验
→ 远程场景：上班时在办公室，用手机/浏览器看 → Vue Web 应用，跨设备
→ 两个场景的交互需求不同：WPF 可以做更复杂的图表操作，Vue 侧重轻量查看+报警
→ 中间层：ASP.NET Core API 统一提供数据（REST 查历史 + SignalR 推实时）
→ 这不是重复建设——是两种不同使用场景
```

### 8.3 部署架构：Windows + Linux 双平台

```
硬约束分析：

  CTP DLL：       Windows Only（C++ 编译的 Windows DLL，无 Linux 版本）
                  → 行情采集 + 交易执行 必须在 Windows 上运行

  .NET 10：       跨平台 ✅（App/Api/Data/Core 全平台运行）

  PostgreSQL：    跨平台 ✅（Linux 上更成熟，Windows 也可）

  WPF：           Windows Only（但是桌面监控，本来就在本机用）
                  → 不是问题——桌面监控不需要跨平台

  Vue Web：       跨平台 ✅（浏览器，任何设备都可以访问）

  SignalR API：   跨平台 ✅（ASP.NET Core 全平台）
```

**两种部署方案**：

```
方案 A — 单机 Windows（推荐，最简单）：
  ┌─────────────────────────────────────┐
  │         Windows Server              │
  │  ┌─────────┐  ┌────────┐  ┌───────┐ │
  │  │ App+Ctp │  │  API   │  │  PG   │ │
  │  │ (交易核心)│  │+SignalR│  │       │ │
  │  └─────────┘  └───┬────┘  └───────┘ │
  │                   │                 │
  │  ┌────────┐       │    ┌──────────┐ │
  │  │  WPF   │───────┘    │ Vue(静态)│ │
  │  │ 桌面端  │            │ 浏览器访问 │ │
  │  └────────┘            └──────────┘ │
  └─────────────────────────────────────┘

  优点：一个机器，零网络延迟
  成本：Windows 云服务器 ≈ 150-200 元/月

方案 B — 分离部署（数据在 Linux，交易在 Windows）：
  ┌──────────────────┐     ┌──────────────────┐
  │  Windows (交易)   │     │  Linux (数据+Web) │
  │  ┌────────────┐  │     │  ┌────────────┐  │
  │  │ App + CTP  │──┼─────┼─→│ PostgreSQL │  │
  │  └────────────┘  │     │  └──────┬─────┘  │
  │                  │     │  ┌───────▼────┐  │
  │                  │     │  │ API+Vue    │  │
  │                  │     │  │ (Web访问)   │  │
  └──────────────────┘     └──────────────────┘

  优点：PG 在 Linux 上更成熟；Web 服务与交易隔离
  缺点：两台机器；跨网络写 PG；CTP 断连时数据丢失风险
```

**当前决策：方案 A — 单机 Windows 部署。**

Linux 部署：可运行 API + PG + Vue（不含 CTP），用于开发/测试/回测场景。不做为实盘部署目标。

### 9.4 多策略运行时架构

**第一性原理推导**：

```
→ 回测和实盘必须用同一份策略代码（Layer 5.2 已推导）
→ 同时运行多个策略，每个可能覆盖多个品种
→ 上班不能看盘 → 运行时调整参数必须通过 API，不能改代码重编译
→ 策略参数 ≠ 风控参数
   风控参数：硬编码，只能在代码里改（保护账户不爆仓）
   策略参数：可在运行时通过 API 调整（如均线周期、阈值、品种选择）
```

**策略生命周期**：

```
[Defined] ──加载──▶ [Loaded] ──配置──▶ [Configured]
                                           │
                              ┌────────────┴────────────┐
                              ▼                         ▼
                         [Running]                 [Standby]
                              │                         │
                    ┌─────────┴─────────┐               │
                    ▼                   ▼               │
               [Paused]           [Error/Stopped]       │
                    │                   │               │
                    └───────────────────┴───────────────┘
                                        ▼
                                    [Unloaded]
```

**策略接口抽象**：

```csharp
// 策略是信号生成器。输入 Tick/Bar → 输出 Signal（不是订单！）
// 订单由 ExecutionService 根据 Signal 生成，经过风控检查
public interface IStrategy
{
    string Id { get; }
    StrategyState State { get; }
    StrategyConfig Config { get; }

    // 生命周期
    Task InitializeAsync(StrategyConfig config);
    Task StartAsync();
    Task PauseAsync();
    Task StopAsync();

    // 数据输入（两个入口，策略按需重写）
    // Tick 级别：每个 tick 触发一次
    void OnTick(Tick tick) { }        // 默认空实现

    // Bar 级别：每个闭合的 Bar 触发一次
    Signal? OnBar(Bar bar) => null;   // 默认空实现

    // 运行时热更新参数
    void UpdateConfig(StrategyConfig newConfig);
}

public record StrategyConfig
{
    public string StrategyId { get; init; }
    public string StrategyType { get; init; }      // "TrendFollowing", "MeanReversion", ...
    public bool IsEnabled { get; init; }
    public string[] Symbols { get; init; }           // 交易品种列表
    public Dictionary<string, string> Parameters { get; init; } // 策略参数 (可运行时修改)
    public RiskLimits RiskLimits { get; init; }      // 该策略的软风控
}
```

**多策略协调 — StrategyManager**：

```
StrategyManager 负责：
  1. 维护所有策略实例的注册表和生命周期
  2. 从 CTP 接收 Tick → 分发给订阅了该品种的策略 → OnTick(Tick)
  3. 从 BarAggregator 接收 Bar → 分发给订阅了该品种的策略 → OnBar(Bar)
  4. 收集所有策略的 Signal → 发给 ExecutionService
  5. 策略间冲突检测（两个策略反向操作同一品种？→ 报警但不阻止）
  6. 总仓位限制（所有策略合计不超过硬风控上限）

数据流：
  CTP/MdProvider ──Tick──→ StrategyManager ──OnTick──→ [Strategy1, Strategy2, ...]
  BarAggregator  ──Bar───→ StrategyManager ──OnBar───→ [Strategy1, Strategy2, ...]
                                   │
              ┌────────────────────┘
              ▼
         Signal[] → ExecutionService → 风控 → CTP TraderApi
```

**什么时候用 OnTick vs OnBar**：

```
中低频策略（主流）：
  → 只重写 OnBar，信号在 Bar 闭合后产生
  → OnTick 保持默认空实现，零开销

需要 Tick 精度的策略：
  → 重写 OnTick，做 tick 级别判断（如突破、大单检测）
  → 可以用 OnBar 做趋势背景判断，OnTick 做精确入场

高频/抢帽子策略：
  → 主要逻辑在 OnTick
  → 但受限于 CTP TCP 延迟（ms 级），不是 μs 级
  → 个人交易者不推荐这个方向
```

**运行时参数热更新**：

```
流程：
  1. 用户通过 WPF 或 Vue → 调用 API: PATCH /api/strategies/{id}/config
  2. API → StrategyManager.UpdateConfig(id, newConfig)
  3. StrategyManager → Strategy.UpdateConfig(newConfig)
  4. 策略在下一个 OnBar 调用时使用新参数
  5. 新配置持久化到 PostgreSQL（重启后恢复）

不需要重启系统。不需要重新编译。
风控参数不受此通道影响（硬编码保护）。
```

**回测与实盘的统一**：

```
同一份 IStrategy 实现：

  实盘模式：
    BarAggregator(实时 CTP tick) → StrategyManager → Strategy.OnBar(Bar)

  回测模式：
    BarReplayer(历史 .tick 文件) → StrategyManager → Strategy.OnBar(Bar)
    区别：信号不发给 ExecutionService，而是发给 VirtualExecutionEngine
    虚拟引擎模拟成交、计算盈亏，不连接 CTP

  关键：Strategy 代码不知道自己是在回测还是实盘——
  它只看到 OnBar(Bar) 调用，返回 Signal。
  这是 Layer 5.2 "同一性" 原则的实现。
```

### 9.5 核心设计原则（来自 CLAUDE.md）

1. **行情与交易物理分离** — MdApi 和 TraderApi 两个独立连接
2. **风控引擎是横切层** — 任何订单到达 CTP 前必须过风控
3. **所有事件必须可回放** — Tick/Order/Trade 全部持久化
4. **合约规格数据驱动** — 保证金/手续费/品种参数不硬编码

## 实战验证：完整交易日数据流推演

> 模拟 2026-06-09（周二）一个完整期货交易日，逐环节验证数据流是否正确。

### 场景设定

```
品种：rb（螺纹钢）— 夜盘 23:00 收盘，有午休
策略：一个基于 1min Bar 的趋势策略，运行中
系统：7×24 运行，当前处于激活态
```

### 08:55 — 日盘开盘前（系统已在运行，从夜盘结束后的待命态切换）

```
┌─ 系统状态 ─────────────────────────────────────────┐
│ 进程：运行中（从未停止）                               │
│ 状态：待命 → 激活（TradingSession 触发）               │
│                                                     │
│ 1. 交易日历确认 2026-06-09 是交易日 ✅                 │
│ 2. TradingSession 判断 09:00 日盘开始                  │
│ 3. CTP MdApi 重连（从昨夜断开后恢复）                   │
│    → 登录成功 → 订阅 rb*, cu*, ...                     │
│ 4. CTP TraderApi 连接（如果有策略在运行）                │
│ 5. 策略确认：从 PG 加载昨日 Bar + 当前持仓              │
│ 6. 健康检查通过 → 等待开盘                             │
└────────────────────────────────────────────────────┘
```

### 09:00:00 — 开盘，第一个 Tick

```
CTP → CtpMdProvider → Channel<TickRecord>

Tick { Symbol="rb2510", LastPrice=3520, Volume=12345, Timestamp=09:00:00.123, ... }

    ├─→ BinaryTickStore.AppendAsync()
    │    → 写入 data/ticks/rb2510/20260609_rb2510.tick
    │    → 异步，不阻塞
    │
    ├─→ BarAggregator.OnTick(tick)
    │    → 内存：rb2510 的 09:00 这个 1min Bar 开始累积
    │    → Open=3520, High=3520, Low=3520, Volume_delta=0
    │
    └─→ StrategyManager.OnTick(tick) → 策略（如果重写了 OnTick）
         → 本策略只关注 OnBar，OnTick 空实现，跳过
```

### 09:00:00 – 09:00:59 — 持续 Tick 流

```
每个 tick 三条路径并行：
  Tick → TickStore (.tick 文件)        ← I/O 线程，不阻塞
  Tick → BarAggregator (更新 High/Low/Close/Volume) ← 内存
  Tick → Strategy.OnTick (空)           ← 无开销

验证点：
  ✅ .tick 写入和策略评估完全解耦——写入慢不影响策略
  ✅ Bar 在内存中累积——策略评估不需要读 .tick 或 PG
  ✅ 全市场 70 合约 × 3 tick/s = 210 tick/s，轻松处理
```


## 性能压力验证：80 品种 × 实时 Tick × 7×24

> 不猜，不算经验值。从每步操作的实际开销计算。

### 输入参数

```
品种数:        80 个
Tick 密度:     Top 20 主力 ≈ 3 tick/s，Mid 30 ≈ 1 tick/s，Tail 30 ≈ 0.3 tick/s
加权平均:      100 tick/s（全市场合计），开盘 burst ≈ 500 tick/s
运行模式:      7×24（但交易仅 ~9.5h/天，其余时间无 tick）
```

### 内存分析：逐项计算

```
1. Tick Channel 缓冲区:
   Channel.CreateBounded<TickRecord>(capacity: 10000)
   TickRecord = 80 bytes (紧凑 struct)
   → 10000 × 80 = 800 KB                                       ≈ 0.8 MB

2. BarAggregator 状态（每个品种维护一个进行中的 Bar）:
   80 品种 × ~200 bytes (OHLCV + L1 depth + tick count)
   → 80 × 200 = 16 KB                                          ≈ 0.02 MB

3. 历史 Bar 内存缓存（策略指标回溯窗口）:
   假设 5 个策略，每个需要回溯 500 个 1min Bar
   1 个 1min Bar = 80 bytes (紧凑 struct)
   → 5 × 80 × 500 × 80 = 16,000,000 bytes                      ≈ 16 MB
   （即使 10 个策略回溯 2000 Bar = 10×80×2000×80 = 128 MB）

4. 策略状态（指标值、信号历史）:
   10 个策略 × 80 品种 × 1 KB = 800 KB                         ≈ 0.8 MB

5. Symbol/Contract 元数据缓存:
   80 Symbol × ~500 bytes + 200 Contract ≈ 140 KB               ≈ 0.1 MB

6. .NET 运行时开销:
   基础运行时 + GC heap + JIT 编译缓存                            ≈ 50 MB

7. PostgreSQL 连接池（Npgsql）:
   10 连接 × ~100 KB = 1 MB                                     ≈ 1 MB

8. SignalR 连接:
   每 Web 客户端 ~50 KB。假设 3 个客户端 = 150 KB               ≈ 0.15 MB

═══════════════════════════════════════════════════════════════════
总计（含 500 Bar 回溯窗口）:                                    ≈ 70 MB
总计（含 2000 Bar 回溯窗口 + 10 策略）:                         ≈ 180 MB
```

**结论：2 GB 内存足够。4 GB 充裕。不存在内存瓶颈。**

### CPU 分析：每条 Tick 的 CPU 开销

```
每条 tick 的处理路径（3 路并行）：

路径 1 — BinaryTickStore.AppendAsync:
  操作: Span<byte> → byte[] → FileStream.WriteAsync
  开销: 非 CPU 密集型。I/O 线程处理，~几微秒 CPU
  → 100 tick/s × 10μs = 1ms CPU/s = 0.1% CPU

路径 2 — BarAggregator.OnTick:
  操作: if(first) Open=Price; High=MAX; Low=MIN; Close=Price; Volume+=delta; TickCount++;
  开销: ~10 CPU 指令，带分支预测
  → 100 tick/s × 10 ops = 1000 ops/s = 0.0001% CPU（无关紧要）

路径 3 — Strategy.OnTick (Bars-only策略 = 空实现):
  开销: 虚方法调用 + 返回 = ~1 ns
  → 0% CPU（编译器可能内联掉）

═══════════════════════════════════════════════════════════════════
每条 tick 的 CPU 总开销: < 0.5%
```

### Bar 闭合时的 CPU 开销（每分钟一次，80 品种）

```
每分钟闭合操作:

1. Bar 生成:
   80 Bar 对象分配 = 80 × 80 bytes = 6.4 KB（在 Gen 0，几乎零开销）

2. 策略评估 (OnBar):
   假设 5 个策略，每个计算 MA(20) + MA(60)
   每次 OnBar ≈ 80 次比较 + 80 次加法（滑动窗口优化后）
   5 × 80 × 80 = 32,000 ops/分钟 = 533 ops/s
   → 0.0...% CPU

3. PG 写入 (异步):
   → I/O 线程，不占用 CPU

4. Bar 缓存更新:
   → 环形缓冲区头指针移动，O(1)

═══════════════════════════════════════════════════════════════════
每分钟 Bar 闭合总 CPU: < 0.1%
```

### 磁盘 I/O 分析

```
.tick 文件写入:
  平均:    100 tick/s × 80 bytes = 8 KB/s
  开盘 burst: 500 tick/s × 80 bytes = 40 KB/s
  全天合计: 8 KB/s × 9.5h ≈ 274 MB/天（所有品种）
  
  → 现代 SSD 顺序写 ≥ 500 MB/s → 使用率 < 0.01%
  → HDD 也轻松胜任（顺序写 ≥ 100 MB/s）

PG Bar 写入:
  80 品种 × 500 bars/天 = 40,000 INSERTs
  分布在 9.5 小时内 = 每秒 1.2 条
  → 完全可以忽略

清理/备份:
  每日清理: 删除 30 天前的 .tick 文件 (约 5-10 GB)
  周末备份: PG dump
  → 安排在待命态（非交易时段），不影响交易
```

### 网络带宽

```

  100 tick/s × ~500 bytes (CTP 原始结构体) = 50 KB/s
  → 家用宽带 (> 1 MB/s) 的 5%

SignalR 推送 (Vue 前端):
  每秒推送: 最新价格 + 最新 Bar
  ~10 条消息/s × 500 bytes = 5 KB/s
  → 无关紧要

═══════════════════════════════════════════════════════════════════
网络: 无压力
```

### 7×24 长期运行的稳定性风险

```
80 品种 × 7×24 连续运行的主要风险不是性能，是稳定性：

风险 1: 内存泄漏
  来源: 事件未退订、StringBuilder/Stream 未释放、Channel Reader 未完成
  检测: 监控 WorkingSet，周度趋势。缓慢增长 > 0.5 MB/天 → 有泄漏
  防御: 定时重启（如每周日凌晨 3:00 自动重启，在待命态重启不影响交易）

风险 2: GC 暂停
  .NET 10 的 Gen 0 GC ≈ 微秒级。Gen 2 GC 可能到毫秒级。
  但堆小（~70 MB），Gen 2 触发频率极低（几天一次）
  → 对中低频策略无影响（毫秒级暂停不影响分钟级信号）

风险 3: 线程池饥饿
  所有 I/O 都是 async（不阻塞线程）
  SignalR 管理自己的线程
  → 饥饿概率 ≈ 0

风险 4: Channel 背压
  Channel 容量 10000 tick。如果消费者慢于生产者 → 背压。
  但消费者（文件写入、Bar 聚合）远快于生产者（100 tick/s）
  → 背压不可能发生。10000 tick buffer = 在 100 tick/s 下 = 100 秒缓冲

风险 5: PostgreSQL 连接泄漏
  使用 Npgsql 连接池，连接自动回收
  但需确保每个 ExecuteAsync 有对应的 using/Dispose
```

### 场景 1：平今 vs 平昨——手续费天差地别

**CFFEX IF 日内交易（平今 = 平昨的 10 倍！）**：

```
09:30 策略信号 → Buy 2 IF2606 @3800
  订单: Open Long, Volume=2
  手续费(开): 3800×300×2×0.000023 = 52.44 元
  保证金: 3800×300×2×12% = 273,600 元

11:00 策略信号 → Sell 2 IF2606 (平仓)
  系统判断: 今日开的仓 → CloseToday
  手续费(平今): 3800×300×2×0.00023 = 524.4 元  ← 开仓的 10 倍！

  总手续费 = 52.44 + 524.4 = 576.84 元/笔
  如果日内交易 2 次/天 = 1153 元/天
  一年 250 天 = 28.8 万元手续费！

如果回测不区分平今/平昨 → 手续费少算 90%
→ 回测年化 30% 的策略，实盘可能只有 8-10%
```

**SHFE rb 螺纹钢（平今免费！相反的逻辑）**：

```
09:00 策略信号 → Buy 5 rb2510 @3520
  手续费(开): 3520×10×5×0.0001 = 17.6 元

14:30 策略信号 → Sell 5 rb2510 (平仓)
  系统判断: 今日开的仓 → CloseToday → rb 平今免！
  手续费(平今): 0 元

  总手续费 = 17.6 元

如果回测按统一费率收 → 多收手续费 → 低估日内策略收益
```

**设计验证**：

```
数据流中的关键判断点：

Strategy.OnBar() → Signal { Direction=Short, Action=Close }
                              │
ExecutionService:             ▼
  1. 查询当前持仓的开仓日期
  2. 查询该品种的 FeeMode:
     ├─ FeeMode.平今免  + OpenDate==Today → CloseToday, 零手续费
     ├─ FeeMode.平今加倍 + OpenDate==Today → CloseToday, N倍手续费
     ├─ FeeMode.双边    + OpenDate==Today → Close(不区分), 标准手续费
     └─ 非今日仓位 → CloseYesterday, 标准手续费
  3. 根据 CommissionRule 计算实际手续费
     → CommissionRule 按 EffectiveDate 匹配最新规则

✅ 设计已覆盖：FeeMode 枚举 + CommissionRule.EffectiveDate
⚠️ 执行层必须正确查询持仓日期并选择 Action 类型
⚠️ 回测引擎必须复制同样的逻辑（Layer 5.2 "同一性"）
```

### 场景 2：不同夜盘时间 → 日线合成规则不同

```
rb (SHFE): 日盘 09:00-15:00 + 夜盘 21:00-23:00
IF (CFFEX): 日盘 09:30-15:00，无夜盘
au (SHFE): 日盘 09:00-15:00 + 夜盘 21:00-02:30

日线合成逻辑：

  rb2510 某个交易日：
    日盘 09:00-15:00 → 375 个 1min bar
    夜盘 21:00-23:00 → 120 个 1min bar
    日线 = 合并 (375+120) 个 bar，TradingDay = 当日

  IF2606 同一个交易日：
    日盘 09:30-15:00 → 330 个 1min bar
    无夜盘
    日线 = 合并 330 个 bar，TradingDay = 当日

  au2612 同一个交易日：
    日盘 09:00-15:00 → 375 个 1min bar
    夜盘 21:00-02:30 → 330 个 1min bar（次日凌晨）
    日线 = 合并 (375+330) 个 bar，TradingDay = 当日

实现：

✅ 设计已覆盖：TradingSession 配置驱动，不硬编码
```

### 规则验证结论

| # | 规则差异 | 设计覆盖 | 缺口 |
|---|---|---|---|
| 1 | 平今/平昨手续费 | FeeMode + CommissionRule + 持仓日期查询 | 执行层需实现 OpenDate 查询逻辑 |
| 2 | 夜盘时间差异 | TradingSession 按品种配置 | 无 |
| 3 | 日线合成（含夜盘） | 从 1min 合并，TradingSession 驱动 | 无 |

### 验证 1：崩溃恢复

```
场景：写入 .tick 文件时断电

文件格式：Header(128B) + N × Record(80B) + Footer(32B)

恢复逻辑：
  1. 文件大小 ＜ 128 → 空文件，删除
  2. 文件大小 ≥ 128 → 读 Header 和 Footer
  3. 实际 RecordCount = (文件大小 - 128 - 32) / RecordSize
  4. 如果 Header.RecordCount ≠ 实际值 → 重建 Footer
  5. CRC32 校验 → 不匹配则重新计算

结论：数据损失 ≤ 最后一个写入中的 tick（最多 1 条）。可恢复。
```

## 结论

从第一性原理推导出的核心设计：

1. **数据是地基** — Tick(.tick 文件) → Bar(PostgreSQL) 自建管道，一个 PG 实例
2. **回测 = 实盘** — 同一份信号代码，不同 Runner
3. **个人系统优化研究速度** — 不是容量，不是延迟
4. **运维极简** — 1 个外部服务（PostgreSQL），其余全在 .NET 进程内
5. **先数据，后策略，最后执行** — 不可逆的依赖顺序
6. **期货优先，股票架构预留** — Market 维度从 Day 1 设计进去

所有决策附带推导链和交叉验证。不包含"因为别人都这么做"或"我熟悉这个工具"的论据。
