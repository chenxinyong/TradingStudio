### 2.2 关键发现

**发现 1：L2-L5 全部为空**
- 本次数据源（金数源）只提供 L1 深度
- 这验证了 TickRecord 80B L1-only 的设计决策
- 中低频策略不需要 L2-L5，即使有也大概率是 0

**发现 2：Volume 和 Turnover 都是累计值（CTP 原始行为）**
- CTP 上报的是当日累计值，每条 tick 的 Volume/Turnover ≥ 上条
- 验证：前 500 条中 Turnover 单调递增，降为 0 次
- 第一条：Volume=282, Turnover=50,238,300 → 最后一条：Volume=647,856, Turnover=114,429,926,300
- 需要自己做增量：`deltaVol = current.Volume - prev.Volume`，注意首条需特殊处理
- 跨日 Volume 会重置（新的交易日从 0 开始）

**发现 2.1：AvgPrice = Turnover / Volume（严格等式）**
- 第一条：50,238,300 / 282 = 178,150.0000 → `当日均价` = 178,150.0000 ✅
- 第二条：84,431,900 / 474 = 178,126.3713 → `当日均价` = 178,126.3713 ✅
- 这个字段是计算值，不需要单独存储，TickRecord 里不存它

**发现 3：TradingDay ≠ ActionDay**
- 夜盘（20:59~01:00）：TradingDay = 20160111，ActionDay = 20160108
- 日盘（09:00~15:21）：TradingDay = 20160111，ActionDay = 20160111
- ActionDay 在夜盘阶段还是前一个业务日期（周五 1/8）
- 这就是为什么 TickRecord 不存合约标识——标识在文件路径 `{tradingDay}_{instrumentId}.tick` 里

**发现 4：持仓量在夜盘和日盘都变化**
- 夜盘开始 OI = 318,604
- 夜盘结束 OI ≈ ?（需要查 row 23528）
- 日盘结束 OI = 356,500
- 整天增仓 37,896 手，说明当天有显著资金流入

**发现 5：收盘后有 15:00~15:21 的 tick**
- 最后一条 tick 在 15:21:03.500
- 这是上期所的收盘后定价时段
- 在交易时段过滤时需要保留或标记这段时间

## 3. 盘口格式：16 字段

从同一数据源的另一种视角：

| 字段 | 值示例 | 说明 |
|------|--------|------|
| 市场代码 | SC | 上期所 |
| 合约代码 | cu1603 | |
| 时间 | 2016-01-08 20:59:00.500 | 完整日期时间（注意：这里的日期是 ActionDay 1/8） |
| 最新 | 35630.0000 | 成交价 |
| 持仓 | 318604 | 总持仓 |
| 增仓 | 140 | OI 变化量（正=增仓，负=减仓） |
| 成交额 | 50238300.0000 | 本笔成交额 |
| 成交量 | 282 | 本笔成交量 |
| 开仓 | 211 | 本笔中开仓量 |
| 平仓 | 71 | 本笔中平仓量 |
| 成交类型 | 多开 | 多开/空开/多平/空平/双平/换手 |
| 方向 | B | B=主动买，S=主动卖 |
| 买一价 | 35620.0000 | |
| 卖一价 | 35630.0000 | |
| 买一量 | 1 | |
| 卖一量 | 51 | |

**与 CTP 格式的关键差异：**

| 维度 | CTP 格式（44 字段） | 盘口格式（16 字段） |
|------|---------------------|---------------------|
| Volume | **累计值**（单调递增，跨日重置） | **每笔增量**（每条独立的成交量） |
| Turnover | **累计值**（单调递增） | **每笔增量**（每条独立的成交额） |
| 时间戳 | UpdateTime + UpdateMillisec（不含日期） | 完整日期时间（含 ActionDay） |
| 开平仓 | 无（需要从 TraderApi 或推导） | 有：多开/空开/多平/空平/双平/换手 |
| 买卖方向 | 无 | 有：B（主动买）/ S（主动卖） |
| OI | 总持仓（累计） | 总持仓 + 增仓（delta） |
| 深度 | L1~L5（本数据源只有 L1） | L1 only |

**数据验证：** 
- CTP 累计增量 = 647,856 - 282 = 647,574 手
- 盘口格式求和 = 647,856 手
- 两者差 282 手 = CTP 首条已含的前序累计量（数据窗口起点前已有 282 手成交）

## 6. 对数据模型设计的验证

### 6.1 ✅ 验证通过的设计

| 设计决策 | 数据证据 |
|----------|----------|
| TickRecord 80B L1-only | L2-L5 在此数据源中全部为零 |
| Price × 10^7 (long) | CTP 价格精度为 .0000（4 位小数），10^7 覆盖 10^6 精度需求绰绰有余 |
| TradingDay ≠ ActionDay | 夜盘 ActionDay 回退到上一交易日 |
| 合约标识走文件路径 | InstrumentID 在整个文件不变；不同的合约是不同的文件 |
| Volume 需要做增量 | CTP 上报的是累计值 |
| 夜盘/日盘分离处理 | 8 小时空档，不能填充 |

### 6.2 ⚠️ 需要关注的点

| 关注点 | 说明 |
|--------|------|
| **收盘后 tick** | 15:00 收盘后还有 15:00~15:21 的 tick（上期所收盘定价），不是错误数据 |
| **集合竞价** | 20:59 第一条是集合竞价撮合结果，Volume 已有值 |
| **涨跌停价** | 35230/33540 在数据中不变，但实际 CTP 可能盘中调整（保证金调整时） |
| **开盘价 = 集合竞价价** | 35630，就是一开盘的 LastPrice，不是独立的字段值变化 |
| **盘口格式时间戳** | 使用 ActionDay（2016-01-08 20:59），不是 TradingDay（20160111），跨系统对接要注意 |

## 7. TickRecord 与 CTP 字段的精确映射

```csharp
// 当 CTP OnRtnDepthMarketData 回调时，映射到我们的 TickRecord：

new TickRecord
{
    ExchangeTimestamp = ParseCtpTimestamp(pData->TradingDay,    // "20160111"
                                          pData->UpdateTime,     // "20:59:00"
                                          pData->UpdateMillisec), // 500
    LocalTimestamp   = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds(),
    
    LastPrice  = (long)(pData->LastPrice * PriceScale),    // 35630.0000 → 356300000000
    BidPrice1  = (long)(pData->BidPrice1 * PriceScale),    // 35620.0000 → 356200000000
    AskPrice1  = (long)(pData->AskPrice1 * PriceScale),    // 35630.0000 → 356300000000
    
    Volume       = pData->Volume,        // long，累计值
    Turnover     = pData->Turnover,      // double
    OpenInterest = pData->OpenInterest,   // double
    
    BidVolume1 = pData->BidVolume1,       // int
    AskVolume1 = pData->AskVolume1,       // int
    
    Flags = (pData->LastPrice >= pData->UpperLimitPrice ? TickRecord.FLAG_UPPER_LIMIT : 0)
          | (pData->LastPrice <= pData->LowerLimitPrice ? TickRecord.FLAG_LOWER_LIMIT : 0)
          | (isAuctionTime ? TickRecord.FLAG_AUCTION : 0)
}
```

```
ActionDay:  1/8 (Fri)  ←──────────夜盘──────────→│←────日盘(1/11 Mon)────→
TradingDay: 20160111  ←──────── 同一个交易日 ────────→

时间轴:
  20:59    21:00       23:00   01:00  [8h空]  09:00     11:30  13:30    15:00  15:21
   │         │           │       │              │         │      │        │      │
   ├─ 集合   ├─ 夜盘活跃  ├─ LME  │              ├─ 日盘最  ├─ 午休 ├─ 下午   ├─ 收盘 ├─ 盘后
   │  竞价   │   期       │  重叠 │              │   活跃   │       │   开盘   │        │   定价
   ▼         ▼           ▼       ▼              ▼         ▼      ▼        ▼      ▼
  Tick 0   Tick 1     Tick N   Tick 23K      Tick 23K+1       Tick 38K  ...   Tick 48K
```

**核心启示：** 
- 一天的 Tick 数据跨越两个日历日（夜盘在周日→周一凌晨）
- TradingDay 统一标识为 20160111
- 文件命名必须用 TradingDay，不能用系统日历日期
- 夜盘和日盘之间的 8 小时空档是正常的，不是数据中断
